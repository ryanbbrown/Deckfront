# Improvements

## What Went Wrong In V1

The V1 experiment process treated strict replay validation as stronger evidence than it really was. The validator became good at checking board-state continuity, movement logs, recruit logs, attacks, healing, upgrades, and win-condition events, but it did not prove that deck turns were generated by the deckbuilding engine.

Several later experiments used one-off `generate-run.mjs` scripts inside `.games/<run>/`. Those scripts produced full-looking replay bundles, but the deck side was usually a scripted summary: predefined hands, played cards, buys, and produced counters. The snapshots often stored only `{ players: [{ id }], activePlayer }`, which passed the loose deck snapshot guard even though it was not a real `GameState`.

The board side was also not an iterative playthrough. In many generated runs, board state was advanced by local helper functions and heuristics inside the experiment folder. That was better than pure hand-authored board snapshots, but it was still custom per-run simulation code rather than a standard game runner with reusable rule enforcement.

The prompts unintentionally allowed this. They required action logs and strict validation, but they did not explicitly require use of the deck CLI or shared engine for every deck turn. They also did not forbid custom run generators. Once `.goals/EXPERIMENTS.md` accepted “deck execution is summarized enough for board-counter accounting,” later experiments inherited that lower standard.

There was also a rules/config conflict: some `board-rules.md` files described a 7-Copper plus draft-coin start, while `deck.yaml` still encoded a fixed starting deck of 5 Copper, 1 Zap, 1 Bandage, and 3 Rest. The process did not force those sources to agree before treating runs as evidence.

## Instruction Improvements

- Define evidence levels more sharply. A run with scripted deck summaries may be useful design sketch evidence, but it must not be called full deck-building evidence.
- Require all full-evidence runs to use shared game/deck infrastructure, not one-off local run generators.
- Forbid custom `.games/<run>/generate-run.mjs` scripts for evidence runs unless the user explicitly approves them and the script uses shared validation-grade APIs.
- Require full deck snapshots before and after every turn: draw, hand, discard, play, trash, supply, phase, resources, active player, and RNG state.
- Require `deck.yaml` and `board-rules.md` to agree on starting deck/setup before any experiment begins.
- Require run notes to state whether the run is full engine evidence, partial/scripted evidence, or invalid.
- Update playtest prompts to say agents must use the deck CLI or approved shared runner for deck actions.
- Update review prompts to inspect deck snapshots and reject minimal placeholder deck states.
- Make “strict validation passed” mean both board legality and deck continuity, or rename weaker modes so they cannot be mistaken for full validation.

## Infrastructure Improvements

- Build a shared playtest runner that combines deck engine actions, board actions, replay writing, and snapshot writing through one path.
- Add reusable board action helpers to `src/` if needed, rather than copying movement/combat/recruit logic into experiment folders.
- Add a real replay recorder around the deck CLI or engine so played cards, buys, trashing, produced counters, and cleanup are captured from actual state transitions.
- Add explicit setup support for drafted starting decks if that remains part of the design. The draft result should be stored in replay metadata and in the first full deck snapshot.
- Add schema validation for persisted deck snapshots instead of the current loose `players + activePlayer` guard.
- Add validation that `entry.deck.drawnHand`, `played`, `bought`, and `produced` match the before/after deck states.
- Add validation that the active player’s hand, play area, discard, draw pile, supply counts, trash, money, actions, buys, and phase transition legally across each deck action.
- Add validation that board-side deck counters cannot appear unless generated by actual played cards in that turn.
- Add tests for invalid minimal deck snapshots so placeholder states fail loudly.
- Add tests for rules/config setup mismatch, especially starting deck mismatch between prose and `deck.yaml`.
- Consider making a single `ExperimentRun` artifact schema that includes ruleset id, map id, setup metadata, evidence level, generator/tool provenance, and validation mode.

## Process Improvements

- Start the reset by fixing validation before running more balance experiments.
- Treat old E019-E024 generated runs as archived design sketches, not valid evidence.
- Keep future experiments small until the runner and validator can prove both deck and board fidelity.
- When subagents are used, give them a concrete command workflow rather than only outcome requirements.
- Require reviewers to inspect provenance: how the run was produced matters, not just whether JSON validates.
